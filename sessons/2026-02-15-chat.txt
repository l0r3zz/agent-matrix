  ────────────────────────────────────────
  Test: kube-proxy-replacement
  Result: false (kube-proxy still active)
  Hypothesis

  The VXLAN tunnels between all three nodes are not passing traffic. Cilium is
  configured for routing-mode: tunnel with tunnel-protocol: vxlan (UDP port
  8472), but every node reports 1/3 reachable. No cross-node pod-to-pod or
  pod-to-service traffic works.

  The most likely cause: a host firewall (iptables/nftables/ufw/cloud security 
  group) is blocking UDP 8472 between the node IPs. Cilium's VXLAN encapsulation
   requires this port to be open on all nodes. Since these appear to be cloud
  VMs with public IPs (144.x, 207.x), a firewall is almost certainly in play.

  A secondary concern: kube-proxy-replacement: false means both kube-proxy
  iptables rules and Cilium are manipulating traffic, which can cause conflicts
  — but the primary blocker right now is the VXLAN tunnel.

  SSH commands I need you to run

  Please run these on all three nodes (or at minimum on k8-0 and k8-2):

  1. Check if VXLAN interface exists and is UP:
  ip -d link show cilium_vxlan

  2. Check host routing table for pod CIDRs:
  ip route | grep -E '10\.200\.[012]'

  3. Test VXLAN port connectivity between nodes (run on k8-0):
  # From k8-0, test UDP 8472 to k8-1 and k8-2:
  nc -uzv 207.244.225.169 8472 -w 3
  nc -uzv 207.244.237.219 8472 -w 3

  4. Check iptables/nftables for DROP rules on UDP 8472:
  iptables -L INPUT -n -v | grep -E '8472|vxlan|DROP'
  iptables -L FORWARD -n -v | grep -E '8472|DROP'
  # Also check nftables and ufw:
  nft list ruleset 2>/dev/null | grep -E '8472|drop|reject' | head -20
  ufw status verbose 2>/dev/null

  5. Check if the cloud provider has a security group/firewall (if applicable):
  # What cloud is this? Check for metadata:
  curl -s --connect-timeout 2 http://169.254.169.254/metadata/v1/ 2>/dev/null |
  head -5
  # Or:
  dmidecode -s system-manufacturer 2>/dev/null

  6. Quick Cilium VXLAN traffic capture (run on k8-0 for ~5 seconds while I 
  trigger a DNS query):
  tcpdump -i eth0 udp port 8472 -c 10 -nn 2>&1 &
  # Then on another terminal, or after:
  tcpdump -i cilium_vxlan -c 10 -nn 2>&1

  The most actionable check is #4 — if you see a firewall dropping UDP 8472,
  that's the fix.

  The Smoking Gun

  VXLAN packets arrive on eth0 but never appear on cilium_vxlan. The tcpdump
  proves it:
  ┌─────────────────┬─────────────────────────────────────────────────┐
  │    Interface    │                Packets captured                 │
  ├─────────────────┼─────────────────────────────────────────────────┤
  │ eth0 (UDP 8472) │ 10 — VXLAN traffic IS flowing between all nodes │
  ├─────────────────┼─────────────────────────────────────────────────┤
  │ cilium_vxlan    │ 0 — inner packets never get delivered           │
  └─────────────────┴─────────────────────────────────────────────────┘
  And here's why — look at UFW on both nodes:

  Default: deny (incoming), allow (outgoing), deny (routed)
                                               ^^^^^^^^^^^^^

  deny (routed) = FORWARD chain policy is DROP. The UFW rules only allow
  specific INPUT ports (6443, 10250, 22, etc.) — there are zero rules for 
  forwarded/routed traffic.

  The nftables confirms it:
  type filter hook forward priority filter; policy drop;

  And k8-2's nftables forward chain has already dropped 4,089 packets (211 KB).

  How this breaks VXLAN

  The packet flow dies at step 5:

  1. Pod on k8-2 sends DNS query to 10.96.0.10 (ClusterIP)
  2. kube-proxy/Cilium DNATs to CoreDNS pod IP 10.200.0.175 (on k8-0)
  3. Cilium encapsulates in VXLAN, sends UDP 8472 to k8-0
  4. k8-0 INPUT chain accepts it (the udp dport 8472 accept rule works)
  5. Kernel decapsulates → inner packet emerges on cilium_vxlan → needs to be
  forwarded to the pod's veth
  6. FORWARD chain → DROP (UFW deny routed, no allow rules)
  7. SYN never reaches CoreDNS. Timeout.

  The existing pods on k8-1 (PostgreSQL, Element) work because they resolved DNS
   and established TCP connections during a window when they happened to start —
   or they're using same-node communication that traverses Cilium BPF hooks
  before hitting netfilter.

  The Fix

  The simplest fix — run on all three nodes:

  ufw route allow in on cilium_vxlan
  ufw route allow in on cilium_host
  ufw route allow in on lxc+

  Or, the more common approach for Kubernetes nodes — just allow all routed
  traffic:

  ufw default allow routed

  Then verify with:
  ufw status verbose | grep -i routed

  After changing UFW, Cilium health should go to 3/3 within ~2 minutes and DNS
  will immediately start working. Want to apply the fix?

● Same-node DNS works! The problem is confirmed to be purely cross-node. And
  look at ufw-before-input — there is NO accept rule for UDP 8472!

  The flow for incoming VXLAN packets through iptables INPUT:
  1. RELATED,ESTABLISHED → no match (packets are UNTRACKED from notrack rule)
  2. DROP INVALID → no match (UNTRACKED ≠ INVALID)
  3. Various ICMP/DHCP rules → no match
  4. No UDP 8472 accept rule → falls through ALL chains
  5. Default policy DROP → packet dropped!

  The nftables udp dport 8472 accept rule we saw earlier is in a separate 
  nftables chain that doesn't prevent the iptables INPUT chain from dropping the
   packet. Both chains evaluate independently — a DROP in either one kills the
  packet.

  then ask you to apply.

  │ chain    │                                                  │ allow routed │

  │ INPUT    │ No ufw allow 8472/udp → iptables INPUT policy    │ ufw allow    │

  Why it was deceptive

  ruleset | grep 8472 showed it counting packets. This made it look like VXLAN
  was being allowed through the firewall. But in nf_tables, the iptables INPUT
  chain evaluates independently — and with no matching rule there, its policy 
  DROP killed the packets regardless of the other chain's ACCEPT.

  3. kube-proxy is redundant — with Cilium as CNI, you could enable
