I need a refresher on how to configure OpenVPN on my DD-WRT router to prepare it to receive incoming connection requests from matrix.v-site.net, basically allowing the synapse process to access 172.23.0.0/16
For starters How do I navigate to the place in the DD-WRT GUI where this is set up?
Here's how to navigate to the OpenVPN configuration area in the DD-WRT GUI, along with guidance on setting it up to accept incoming connections from your Matrix/Synapse server.

Navigating to OpenVPN in DD-WRT
Open a browser and go to your DD-WRT router's LAN IP address (typically http://192.168.1.1).
​

Click the Services tab at the top of the control panel.
​

Click the VPN sub-tab.
​

In the OpenVPN Daemon section (this is the server section, not the client), click Enable. This expands all the server configuration fields.
​

Important: You want the OpenVPN Daemon/Server section — not the "OpenVPN Client" section. The server section is what allows your router to receive incoming VPN connections from remote hosts like matrix.v-site.net.
​

Key Configuration Steps
Once the OpenVPN Daemon section is expanded, you'll see fields for certificates, keys, and a freeform config box. Here's what matters for your use case:

Certificates and Keys
You need to paste the following into their respective fields:
​

CA Cert — your Certificate Authority certificate (ca.crt)

Public Server Cert — the server certificate (server.crt)

Private Server Key — the server private key (server.key)

DH PEM — the Diffie-Hellman parameters (dh.pem)

OpenVPN Config Box
In the OpenVPN Config text area, you'll define the server behavior. For your scenario — allowing the Synapse process on matrix.v-site.net to reach the 172.23.0.0/16 subnet — pay attention to these directives:

text
server 10.8.0.0 255.255.255.0
proto udp
port 1194
dev tun0
keepalive 10 120
persist-key
persist-tun
push "route 172.23.0.0 255.255.0.0"
verb 3
The critical line is push "route 172.23.0.0 255.255.0.0" — this tells the connecting client (matrix.v-site.net) that traffic destined for 172.23.0.0/16 should be routed through the VPN tunnel.
​

Firewall Rules
After saving the VPN config, go to Administration → Commands and add firewall rules so VPN traffic can flow to your local subnet:
​

bash
iptables -t nat -A POSTROUTING -s 10.8.0.0/24 -j MASQUERADE
iptables -I FORWARD -p udp -s 10.8.0.0/24 -j ACCEPT
iptables -I INPUT -p udp --dport=1194 -j ACCEPT
iptables -I FORWARD -i tun0 -o br0 -j ACCEPT
iptables -I FORWARD -i br0 -o tun0 -j ACCEPT
Click Save Firewall to persist these rules across reboots.
​

Checking Connection Status
Once matrix.v-site.net connects with its client certificate, you can verify the tunnel by going to Status → OpenVPN in the DD-WRT GUI. A successful connection will show "CONNECTED SUCCESS" along with the assigned VPN IP address.
​

After this is working, the Synapse process on the remote server will be able to route traffic to any host on 172.23.0.0/16 through the VPN tunnel, as long as the hosts on that subnet know to route return traffic back through the DD-WRT router.

Here's a complete walkthrough for generating all the OpenVPN certificates and keys you need using Smallstep's step CLI — no running CA server required. You'll use step certificate create in offline/standalone mode to build a full PKI, then paste the resulting PEM files into your DD-WRT router.

Prerequisites
Install the step CLI tool. On macOS:

bash
brew install step
On Linux (Debian/Ubuntu):

bash
wget https://dl.smallstep.com/gh-release/cli/docs-cli-install/v0.28.0/step-cli_0.28.0_amd64.deb
sudo dpkg -i step-cli_0.28.0_amd64.deb
You can also download binaries from the Smallstep GitHub releases.
​

Step 1 — Create a Root CA
This generates a self-signed root certificate and key. You'll be prompted for a password to encrypt the private key:

bash
step certificate create --profile root-ca \
  "My VPN Root CA" root-ca.crt root-ca.key
Keep root-ca.key safe and offline after you're done issuing certificates. You'll only need root-ca.crt on both the server (DD-WRT) and client (matrix.v-site.net).
​

Step 2 — Create the Server Certificate (for DD-WRT)
Generate a leaf certificate signed by your root CA. Use your router's WAN IP or a DNS name as the subject, and add any SANs you need. OpenVPN typically expects RSA keys, so specify --kty RSA:
​

bash
step certificate create "ddwrt-vpn-server" server.crt server.key \
  --profile leaf \
  --ca ./root-ca.crt \
  --ca-key ./root-ca.key \
  --san your-router-wan-ip-or-hostname \
  --kty RSA --size 2048 \
  --not-after 8760h \
  --no-password --insecure
Key flags explained:

--profile leaf — generates a standard TLS leaf certificate with serverAuth and clientAuth extended key usage
​

--san — adds Subject Alternative Names (IP or DNS); use multiple --san flags if needed
​

--not-after 8760h — sets validity to 1 year (adjust as needed)
​

--no-password --insecure — writes the private key unencrypted, which DD-WRT requires since it can't prompt for a passphrase at boot
​

Step 3 — Create the Client Certificate (for matrix.v-site.net)
bash
step certificate create "matrix.v-site.net" client.crt client.key \
  --profile leaf \
  --ca ./root-ca.crt \
  --ca-key ./root-ca.key \
  --san matrix.v-site.net \
  --kty RSA --size 2048 \
  --not-after 8760h \
  --no-password --insecure
Each client that connects should have its own unique certificate with a unique Common Name.
​

Step 4 — Generate Diffie-Hellman Parameters
The step CLI doesn't generate DH parameters directly, so use OpenSSL for this one:

bash
openssl dhparam -out dh2048.pem 2048
This can take a few minutes. The DH parameters are not secret and are needed by the OpenVPN server only.
​

Step 5 — (Optional) Generate a TLS-Auth Key
For additional HMAC security on the control channel:

bash
openvpn --genkey secret ta.key
Summary of Generated Files
File	Where It Goes	Secret?
root-ca.crt	DD-WRT CA Cert field + client config	No
root-ca.key	Offline storage only	Yes
server.crt	DD-WRT Public Server Cert field	No
server.key	DD-WRT Private Server Key field	Yes
client.crt	Client .ovpn config on matrix.v-site.net	No
client.key	Client .ovpn config on matrix.v-site.net	Yes
dh2048.pem	DD-WRT DH PEM field	No
ta.key (optional)	Both server and client	Yes
Step 6 — Paste into DD-WRT
Navigate to Services → VPN → OpenVPN Daemon (as discussed previously), and paste the contents of each PEM file into the corresponding field:
​

CA Cert → contents of root-ca.crt

Public Server Cert → contents of server.crt

Private Server Key → contents of server.key

DH PEM → contents of dh2048.pem

Use cat to display each file and copy the full output including the -----BEGIN ...----- and -----END ...----- lines.

Verifying a Certificate
You can inspect any certificate you created:

bash
step certificate inspect server.crt --short
Or verify the chain:

bash
step certificate verify server.crt --roots root-ca.crt
No output means the certificate is valid and properly signed by your root CA.
​

Why step Instead of easy-rsa?
The step CLI is a modern alternative to easy-rsa with a cleaner interface and more flexibility. The key advantages:
​

Single binary — no directory of shell scripts to manage

Explicit profiles — --profile root-ca, intermediate-ca, leaf make intent clear

Template support — customizable certificate fields via JSON templates if you ever need non-standard extensions
​

Composable — works both standalone (as shown here) and as a client to a full step-ca server if you later want automated certificate lifecycle management
​
